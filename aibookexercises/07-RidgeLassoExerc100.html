2 Unregularized Benchmark Model

As a benchmark, we begin with an unregularized model. In this case, a multivariate Degree-2 Polynomial Model. It includes the variables \(Sqft\), \(Bedr\), and \(Year\). They are used in their original form, squared, and as various interaction terms:

\[\begin{eqnarray} Price&=& \beta_1 Sqft+\beta_2 Bedr+\beta_3 Year\nonumber \\ &+&\beta_4 Sqft^2+\beta_5 Bedr^2+\beta_6 Year^2\nonumber \\ &+&\beta_7 Sqft Bedr+\beta_8 Sqft Year+\beta_9 Bedr Year\nonumber \\ &+&\beta_{10} Sqft^2 Bedr+\beta_{11} Sqft Bedr^2+\beta_{12} Sqft^2 Bedr^2\nonumber \\ &+&\beta_{13} Sqft^2 Year+\beta_{14} Sqft Year^2+\beta_{15} Sqft^2 Year^2\nonumber \\ &+&\beta_{16} Bedr^2 Year+\beta_{17} Bedr Year^2+\beta_{18} Bedr^2 Year^2 \end{eqnarray}\]

NoteWarning

In the code block above, you changed hyper-parameters and evaluated the effects based on the testing data to better understand regularization. For educational experiments that is perfectly OK.

However, for real research never use the testing data to find the best hyper-parameters. Otherwise, you end up with a model that is highly specialized to the testing data, but when used on new data in the production phase, it would likely perform very poorly — a form of overfitting.

Instead, use validation data derived from the training dataset to evaluate the performance of hyper-parameters. This is the underlying principle of tuning, and in the next section, you will tune the workflow model to improve performance.

Step 5 - Create a Hyper-Parameter Grid:

Previously, when only one hyper-parameter was tuned, you added the values to be tried out in a data frame column named with the same name as the hyper-parameter. When you have more than one hyper-parameter, it gets a little more complicated because you want to try out different combinations of hyper-parameters.

If you decide to try out all combinations of the values from two or more hyper-parameters, you can use the expand.grid() command. The arguments of the expand.grid() command are the hyper-parameters you would like to try out. The assigned values to these arguments are the values you want to try out for each of the hyper-parameters. They are assigned to each argument as R vector objects. The expand.grid() command then generates all possible combinations for the values of the hyper-parameters.

For example, if you leave the code below unchanged (recommended for the first trial) you combine 8 penalty values with three mixture values, and you can use the expand.grid() command to generate 24 possible combinations (\(8 \cdot 3\)):

Hint: Leave the values unchanged for the first trial, but then use other values for subsequent trials. Beware, if the number of combination increases the execution time increases also in Step 7 – Step 10.

The R vector objects assigned to penalty= and mixture= are created with c() commands by listing the related values.

You can see in the printout above that expand.grid() generates all possible combinations of the penalty= values and the mixture= values. For the default values each of these 24 combinations will be tried out when you tune the workflow later on in Step 7 – Step 10.

The expand.grid() command is very helpful when you combine multiple hyper-parameters with several values each. Imagine you want to create a grid with all combinations of five hyper-parameters with ten values each, expand.grid() would create \(100{,}000\) parameter combinations (\(10\cdot 10\cdot 10\cdot 10\cdot 10=10^5\)) for you. Be advised, tuning \(100{,}000\) hyper-parameter combinations would use a lot of computing time.

In general, computing time increases exponentially with the number of hyper-parameters, while computing time increases proportionally when the values for each hyper-parameter increase. These facts must be considered when deciding how many hyper-parameters you want to tune and how many values you like to consider for each hyper-parameter.
